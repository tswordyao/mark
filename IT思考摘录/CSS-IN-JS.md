css in  js

```
前端关注点分离的概念，刚刚给落后了好几年的jsp开发人员灌输好，让他们规范起来。css in js , 甚至 html in js 又来了。

把html和js和css三者按类型分离存放，还是按组件把相关的html，css，js同一存放？

就像把同一模块或业务页面的html，cs，js统一存放一个功能目录，还是放置在各自的类型目录一样， 各有利弊。

```

css in js有很多好处。
```
一来，简单的插件不在需要额外的css文件，而是直接写在js里就可以，引用插件时就会简洁，不需要考虑相对样式路径。

二来，可以利用js的逻辑和变量机制，实现动态的“class object”和值。

三来，更容易控制样式污染。样式命名和应用规范，优先级或叠加冲突，在项目大了以后，其实也非常难管理。它比js污染引起的问题严重性低些，但维护更琐碎，更纠缠。
```


另一方面：
```
1，如果插件或组件依赖的不止于样式，还有图片等其他资源，那么css in js 并不能实现all in，除非图片足够微量，能用base64替代。

2，原先切换class，组合class来实现的样式操作习惯需要改变，而这里其实有很多优点，css in js无法办到。比如切换一个父容器的class可以轻易改变内部所有子元素对应的状态，以实现整套各处不同的style。css obj只能针对单个元素，子元素影响只能由js控制批量替换。这个性能和算法都不如原生css的应用。

3，原生css中关于active，focus，hover等状态伪类的实现，其实是依靠浏览器自身的监听机制。css in js用js事件模拟则相当笨重，甚至根本难以实现。

4，从性能方面考虑，动画，过渡等只能由css，js动态生成style拼入head中，跟原生css区别就不大了，也不利于频繁动态修改。

5，覆盖样式的困难
```

```
我认为，应该把组件的基础样式和可定制样式分离考虑。
基础样式比如容器的clearfix，定位，overflow处理等规则样式，是组件脱离了就不能正常使用的。
 而可定制样式为fontsize，color，shadow等，是可以根据需求变化的。
```
